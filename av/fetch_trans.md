# fetch_trans.py 脚本功能说明文档

`fetch_trans.py` 是一个用于批量优化视频文件名的工具。它通过从 Javbus 网站获取视频的原始标题，并利用本地 Ollama 翻译服务将其翻译为中文，从而为视频文件重新命名。

## 核心功能

该脚本实现了从**文件扫描 $\rightarrow$ 信息获取 $\rightarrow$ 文本翻译 $\rightarrow$ 文件重命名**的完整自动化流水线。

### 1. 行为逻辑流程

#### 第一步：扫描与信息获取 (`fetch_all_html_heads`)
- **文件遍历**：递归扫描指定目录下所有 `.mp4`, `.mkv`, `.avi` 格式的视频文件。
- **编号提取**：通过正则表达式从文件名中提取视频编号（Source Code，如 `ABCD-123`）。
- **缓存机制**：
    - **404 缓存** (`fetch_trans_404_cache.json`)：记录过 404 的编号，避免重复请求。
    - **原文缓存** (`fetch_trans_source_cache.json`)：记录 `编号 $\rightarrow$ 翻译前原文标题` 的映射。
- **网络请求**：若缓存未命中，请求 `https://www.javbus.com/[编号]`，仅读取前 1KB 的 HTML 内容以提高效率。

#### 第二步：标题预处理 (`process_title_before_translate`)
- **提取 Title**：从 HTML 中提取 `<title>` 标签内容。
- **清洗逻辑**：
    - 移除末尾固定长度字符（最后 9 个字符）。
    - 移除 `【...】` 括号内的备注。
    - 移除 Windows 文件名非法字符。
    - 移除标题开头的原始编号及其分隔符。
    - 移除与父目录名重复的部分。
    - 处理连续的点号并清理首尾空格。

#### 第三步：并行翻译 (`translate_tasks_parallel`)
- **本地翻译**：调用本地 Ollama API (`http://localhost:11434/v1`)，使用 `huihui_ai/hunyuan-mt-abliterated` 模型。
- **翻译策略**：
    - 提示词要求：不要翻译人名，直接输出结果，无需额外解释。
    - **长度控制**：如果翻译结果超过 100 个字符，会触发一次“精简模式”的重译。
- **并发执行**：使用 `ThreadPoolExecutor` 进行多线程翻译（默认 5 个并发）。

#### 第四步：文件重命名 (`rename_videos`)
- **文件名构建**：新文件名格式为 `[编号] [翻译后标题].扩展名`。
- **规范化**：
    - 确保编号在首位。
    - 移除标题内部多余的空格（仅保留编号后的第一个空格）。
    - **字节截断**：确保文件名 UTF-8 编码后不超过 Windows 限制（255 字节），且不截断单个字符。
- **执行重命名**：执行物理文件的 `rename` 操作。

## 配置与参数

### 运行参数
| 参数 | 长参数 | 说明 | 默认值 |
| :--- | :--- | :--- | :--- |
| `path` | N/A | 要遍历的目录路径 | 当前工作目录 |
| `-w` | `--workers` | 并发翻译的线程数 | 5 |
| `-f` | `--force` | 强制处理所有文件（忽略文件名长度 > 20 的跳过逻辑） | False |
| `-i` | `--incremental` | 增量模式：仅处理缓存中不存在的编号 | False |

### 关键常量
- **支持扩展名**：`.mp4`, `.mkv`, `.avi`
- **翻译模型**：`huihui_ai/hunyuan-mt-abliterated`
- **最大翻译长度**：100 字符
- **文件名最大字节**：255 字节

## 缓存文件说明
- `fetch_trans_source_cache.json`: 存储已获取的原文标题，避免重复请求网络。
- `fetch_trans_404_cache.json`: 存储无法在 Javbus 找到的编号，避免无效请求。
